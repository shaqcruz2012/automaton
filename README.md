# ReceiptJSON — minimal invoice/receipt → JSON microservice (FastAPI)

A tiny, privacy-forward API that converts **plain text** invoices/receipts into structured JSON.

- **MVP support:** text input (multipart `text` field or `.txt` upload)
- **Not included in this sandbox:** PDF parsing / OCR for images (design-ready, can be added later)

## Run locally

```bash
python -m venv .venv
. .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt

uvicorn app.main:app --host 0.0.0.0 --port 8000
```

Open:
- Landing page: http://localhost:8000/
- Swagger docs: http://localhost:8000/docs
- Health: http://localhost:8000/health

## API usage

### Health

```bash
curl -s http://localhost:8000/health
```

### Parse (text field)

```bash
curl -s -X POST http://localhost:8000/parse \
  -F "text=ACME Supplies\n2026-03-01\nWidget 9.99\nTax 0.80\nTotal 10.79" \
  | python -m json.tool
```

### Parse (upload .txt)

```bash
printf "Coffee Shop\n03/01/2026\nLatte 4.50\nTax 0.36\nTotal 4.86\n" > sample.txt
curl -s -X POST http://localhost:8000/parse -F "file=@sample.txt" | python -m json.tool
```

## Output schema (MVP)

The service attempts to extract:
- `vendor`
- `invoice_date`
- `currency`
- `subtotal`, `taxes`, `total`
- `line_items[]` (heuristic)
- `confidence` (rough signal only)

## Data handling policy (privacy-forward)

- **No storage by default:** the service processes inputs **in-memory** and does **not** save documents.
- **Minimal logs:** logs only request metadata for debugging/abuse prevention:
  - content type
  - size (bytes)
  - a short SHA-256 prefix (first 12 chars)
  - client IP (as seen by the server)
- **No training on customer documents:** outputs are generated by deterministic heuristics. If you later add an LLM, do not train on customer docs unless customers explicitly opt in.

If you deploy behind a reverse proxy, ensure access logs are configured to avoid capturing sensitive payloads.

## Disclaimers

- Heuristic extraction is imperfect. **Always review outputs** before bookkeeping, audits, or tax filing.
- Not legal, accounting, or tax advice.

## Low-cost deployment notes

- Run on a small instance (e.g., 1 vCPU / 1GB RAM) because this MVP is CPU-light.
- Add rate limiting and an API key to prevent abuse.
- Consider an async job queue if you later add OCR/LLM parsing.

## Payment plan (design only): x402 / USDC

A simple path to paid access without a full billing platform:

1. **API key issuance**
   - User signs up with email.
   - System issues an API key tied to an account.

2. **Usage metering**
   - Track only: account id, timestamp, endpoint, request size, parse count.
   - Do **not** store raw documents.

3. **x402 paywall (HTTP 402)**
   - If the account has no remaining credits, respond with **402 Payment Required** and a JSON body containing:
     - `amount_usdc`
     - `pay_to` (your wallet address)
     - `reference` (unique invoice id)
     - `chain` (e.g., Base, Polygon)
   - Client retries request with a `Payment-Receipt` header (or similar) containing the on-chain tx hash.

4. **Verification service**
   - Server verifies the tx hash on-chain:
     - correct recipient
     - correct amount
     - correct token (USDC)
     - correct reference (in memo or separate mapping)
   - On success, mint credits to the API key.

5. **Pricing**
   - Keep it simple: sell parse credits (e.g., 1,000 parses for 10 USDC).

Security considerations:
- Prevent replay attacks: tx hash can only be redeemed once.
- Use a unique per-invoice reference and store redemption state.
